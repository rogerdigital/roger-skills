Create a pull request for the current branch. Base branch: `$ARGUMENTS` (defaults to `main`).

## Process

### 1. Analyze branch state

```bash
git status
git branch --show-current
git log main..HEAD --oneline
git diff main...HEAD --stat
```

If `$ARGUMENTS` specifies a base branch other than main, use that instead.

If there are uncommitted changes, warn the user and stop.

If the branch has no commits ahead of base, report clearly and stop.

### 2. Understand the changes

Read the full diff to understand what was changed and why:

```bash
git diff main...HEAD
```

Identify:
- The main purpose of the branch
- Key files changed and what they do
- Whether this is a feature, bugfix, refactor, or chore
- Any breaking changes

### 3. Write the PR title

- Under 72 characters
- Use Conventional Commits prefix: `feat:`, `fix:`, `refactor:`, etc.
- Imperative mood, no trailing period
- Summarize the "what" not the "how"

### 4. Write the PR body

Structure the body as:

```markdown
## Summary
<2-3 bullet points capturing the key changes>

## Changes
- <specific change 1>
- <specific change 2>
- ...

## Test plan
- [ ] <manual or automated verification step 1>
- [ ] <verification step 2>
- [ ] <edge case to check>
```

Rules for the body:
- Summary focuses on user-visible impact
- Changes list is specific and traceable to files
- Test plan includes concrete verification steps, not just "ran tests"
- If there are breaking changes, add a `## Breaking changes` section with migration notes

### 5. Push and create

If the branch doesn't have a remote tracking branch:

```bash
git push -u origin <branch-name>
```

Create the PR:

```bash
gh pr create --base <base-branch> --title "<title>" --body "$(cat <<'EOF'
<body>
EOF
)"
```

If there are linked issues, add `--fixes #123` or mention in the body.

## Rules

- Do NOT include AI tool attributions, co-author lines, or "Generated by" footers.
- Do NOT force push (`--force`).
- Match the PR to the scope of the branch — don't over-describe minor changes.
